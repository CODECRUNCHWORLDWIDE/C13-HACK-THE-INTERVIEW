# Week 3 — Quiz

Ten questions. Lectures closed.

**Q1.** A recruiter using LinkedIn Recruiter searches `backend engineer Python AWS`. Which of the following profiles is most likely to appear in the top results?

- A) "Software Engineer | Passionate about technology | Lifelong learner"
- B) "Senior Backend Engineer | Python, PostgreSQL, AWS | Payments infra at Acme"
- C) "Father, husband, engineer, runner"
- D) "Looking for opportunities | Available immediately"

---

**Q2.** Your resume lists "Senior Backend Engineer" at Acme; your LinkedIn lists "Backend Engineer II" at the same company. A recruiter pulls both. The most likely conclusion is:

- A) The recruiter doesn't notice — they only read one surface.
- B) The recruiter assumes the resume is inflated and treats the LinkedIn as the source of truth.
- C) LinkedIn automatically corrects mismatches before recruiters see them.
- D) The mismatch helps because it shows growth over time.

---

**Q3.** The "Open to Work" feature on LinkedIn has two modes. Which statement is correct?

- A) The public ring mode is always preferable because it maximizes recruiter inbound.
- B) The recruiter-only / discreet mode is invisible to your colleagues but still surfaces you in recruiter-side active-candidate filters.
- C) Both modes are functionally identical; only the visual styling differs.
- D) Turning on either mode prevents you from receiving cold outreach.

**Q4.** What is a GitHub "profile README"?

- A) A pinned post on your activity feed.
- B) A `README.md` in a repository named exactly your username, which renders at the top of your GitHub profile.
- C) An auto-generated summary GitHub creates from your activity.
- D) A premium feature available only with GitHub Pro.

**Q5.** A hiring manager typically spends roughly how long on a candidate's GitHub profile before the screen?

- A) 5-10 seconds.
- B) 60-90 seconds, plus another 60-90 seconds inside one pinned repo if something catches their eye.
- C) 15-20 minutes, reading code line by line.
- D) Over an hour, including running the code.

**Q6.** Which of the following is a load-bearing element to include in a project README, per Lecture 2?

- A) Your full personal biography.
- B) A "Quick start" section with three commands or fewer to get the project running.
- C) A list of every technology you've ever used.
- D) A long list of acknowledgments.

**Q7.** Your GitHub profile shows 73 repositories, 60 of which are forks of tutorials. The pinned repos are good, but the long list dilutes them. The recommended fix is:

- A) Pin all 6 slots with the best repos to compete with the volume.
- B) Delete, archive, or make private the tutorial forks; keep only repos worth visible signal.
- C) Add 20 more repos so the forks don't stand out.
- D) Leave it; recruiters skip the repo list anyway.

---

**Q8.** Your contribution grid is mostly gray (most of the year shows no activity), but you have a strong day job whose code is in a private corporate repo. The recommended response per Lecture 2 is:

- A) Fake activity by force-pushing dummy commits to inflate the grid.
- B) Either contribute something recent to a public project, or update your profile README to point readers at where your work actually lives.
- C) Make all your activity private so the grid disappears entirely.
- D) Switch to GitLab.

---

**Q9.** Which sentence is best as the opening of a LinkedIn About section?

- A) "I am a passionate software engineer who loves learning new technologies and solving complex problems."
- B) "I'm a backend engineer at Acme; I own the payments service that processes ~$2M/day across 12 countries (Python/Go on AWS)."
- C) "Welcome to my LinkedIn profile! Thanks for visiting."
- D) "Self-starter and team player with strong communication skills."

---

**Q10.** You have six pinned-repo slots on GitHub. Lecture 2 recommends pinning approximately:

- A) Six, always — fill every slot for maximum coverage.
- B) Three, chosen intentionally to demonstrate distinct skills (main project, breadth, polish).
- C) One — pin only your single best project.
- D) Zero — let GitHub auto-surface based on activity.

---

## Answer key

<details>
<summary>Reveal after attempting</summary>

1. **B** — Headline B contains the specific role, the technology nouns (Python, AWS, PostgreSQL), and the domain (payments). The others contain no searchable engineering nouns.
2. **B** — Recruiters routinely cross-check resume against LinkedIn. The mismatch reads as either carelessness or inflation; the LinkedIn (with the lower title) often becomes the assumed truth. Fix by aligning both surfaces.
3. **B** — Recruiter-only mode is the low-cost upside: surfaces you in active-candidate filters, invisible to non-Recruiter viewers. Public mode has additional inbound effects but signals to colleagues you're searching.
4. **B** — A repo named exactly your username, containing a `README.md`, renders that README on your profile page above the pinned repos.
5. **B** — 60-90 seconds on the profile, 60-90 seconds in one pinned repo if anything draws them in. Sub-3 minutes total. The implications shape what surfaces you optimize.
6. **B** — The Quick start (three commands or fewer to a running project) is one of the most load-bearing sections of a project README. A README without it is harder to evaluate.
7. **B** — Fork graveyards dilute curated pins. Delete, archive, or make private the tutorial forks. The visible repo list should reflect curation.
8. **B** — Either fix the grid by contributing something honest, or contextualize it via the profile README. Faking activity is widely recognized and counterproductive.
9. **B** — Opens with the current role, the employer, a specific quantified outcome, and a technology stack. The others contain no specific nouns or use phrases recruiters discount ("passionate," "self-starter," welcome banners).
10. **B** — Three intentional pins beat six mediocre ones. Lecture 2's mix: one main project, one breadth repo, one polish repo.

</details>

If under 7, re-read the lecture you missed. If 9+, you're ready for the homework.
